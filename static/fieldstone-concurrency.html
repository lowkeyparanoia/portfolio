Why Go is the right language for this

One binary, zero runtime

Go compiles to a single static executable — no Node/Python runtime to install, no node_modules. docker build → a ~25MB image. Trivial to self-host.

Concurrency is built-in

Goroutines + channels are language primitives, not a library. A BaaS is inherently concurrent (many sockets, jobs, requests at once) — Go makes that cheap and safe.

Fast + predictable

Compiled, GC'd, great at I/O concurrency. Handles thousands of live WebSocket clients on modest hardware — exactly a realtime BaaS's job.

Goroutines — concurrency that's nearly free

A goroutine is a function running concurrently, scheduled by the Go runtime — not the OS. You start one with the keyword go. They're tiny (~2KB stack that grows on demand) vs an OS thread (~1MB), so you can run hundreds of thousands. Fieldstone spawns two per WebSocket client.

// internal/realtime/manager.go — two goroutines per connected client
func (m *Manager) HandleWebSocket(w http.ResponseWriter, r *http.Request) {
  conn, _ := m.upgrader.Upgrade(w, r, nil)
  client := &Client{ ID: generateClientID(), Conn: conn, Send: make(chan []byte, 256) }
  m.register <- client
  go client.readPump()    // goroutine 1: socket → hub
  go client.writePump()   // goroutine 2: hub → socket
}

The runtime scheduler (GMP), briefly

Go multiplexes many Goroutines onto a few OS threads (M) via logical Processors (one per CPU core). When a goroutine blocks on I/O (a socket read), the scheduler parks it and runs another on the same thread — so a blocked client never wastes a thread. That's why one Fieldstone process can hold thousands of idle WebSocket connections cheaply.

The nuance: go f() doesn't mean "run on another CPU now" — it means "hand this to the scheduler to run concurrently." Concurrency (dealing with many things) ≠ parallelism (doing many things at once); Go gives you concurrency, and uses your cores for parallelism when it helps.

Channels & select — the realtime hub

Goroutines don't share memory by locking it — they communicate by passing messages over channels (Go's motto: "share memory by communicating"). A channel is a typed pipe; select waits on several at once. Fieldstone's realtime hub is the textbook example: one goroutine owns all the state and listens on three channels.

// internal/realtime/manager.go — the hub's single event loop
func (m *Manager) Run() {
  for {
    select {
    case client := <-m.register:    m.clients[client.ID] = client          // new socket
    case client := <-m.unregister:  delete(m.clients, client.ID); close(client.Send)
    case event  := <-m.broadcast:   m.broadcastToRoom(event)            // fan-out
    }
  }
}

The pro move: never let a slow client block the hub

// non-blocking send — the heart of a robust hub
for _, client := range room {
  select {
  case client.Send <- data:        // fast client: delivered
  default:                         // buffer full (slow client): DON'T wait —
    close(client.Send); m.unregister <- client   // drop it, protect everyone else
  }
}

Tick the "🐢 buffer full" box and broadcast again: the slow client is dropped instead of freezing the whole hub. The buffered channels (broadcast 256, each client's Send 256) absorb bursts; the default case is the release valve. This one pattern is why a single hub goroutine can safely serve thousands of clients.

Buffered vs unbuffered: make(chan T) blocks until both sides are ready (a handshake). make(chan T, 256) lets the sender keep going until 256 are queued. Fieldstone buffers the hot paths so a momentary spike doesn't stall senders.

Synchronization — RWMutex, semaphores, WaitGroup, context

Channels coordinate flow; mutexes protect shared state; semaphores bound concurrency; WaitGroups wait for all; context cancels. Fieldstone uses all five. Here are the two best demos.

RWMutex — many readers, one writer

The hub's clients/rooms maps are read constantly (every broadcast) and written rarely (connect/disconnect). A sync.RWMutex lets unlimited readers in parallel, but a writer gets exclusive access. (Fieldstone uses RWMutex in 11 places; RLock for reads, Lock for writes.)

Readers run together. A writer makes everyone else wait.

Semaphore worker pool — bounded concurrency

The job queue runs at most N jobs at once using a channel as a semaphore (sem chan struct{} of size N). Acquire a slot before working, release on done. Extra jobs wait. This is how Fieldstone keeps background work from swamping the box.

// internal/jobs/queue.go — semaphore + goroutine per job
select {
case w.sem <- struct{}{}:        // got a slot (sem has capacity N)
default:  return                  // pool full → skip this tick
}
job, _ := w.db.FetchAndLockJob(ctx, w.queue, w.id)   // atomically claim a job
go func() { defer func(){ <-w.sem }(); w.executeJob(ctx, job) }()  // release on done

WaitGroup + context — graceful shutdown

// wait for all workers, but give up after a timeout (context)
done := make(chan struct{}{})
go func(){ q.wg.Wait(); close(done) }()   // WaitGroup: all workers Done()
select {
case <-done:        // everyone finished cleanly
case <-ctx.Done(): // deadline hit → stop waiting (context cancellation)
}

context.Context appears 285 times across the codebase — it's the cancellation/deadline signal threaded through every request and job, so when a client disconnects or a timeout fires, the whole chain of goroutines below it stops instead of leaking.

The throughline: a BaaS is "do many independent things at once safely." Fieldstone answers that with the smallest toolkit — a goroutine per unit of work, channels to move data between them, a mutex where state is genuinely shared, and context to cancel it all. No framework magic; just Go.

vs PocketBase — same idea, wider surface

PocketBase is the reference Go single-binary BaaS (SQLite, auth, realtime, file storage, admin UI, JS-VM hooks). It's excellent and very mature. Fieldstone takes the same shape and widens the API + extensibility surface. Honest comparison:

The honest read: PocketBase wins on maturity, community, and "it just works" polish — if you want a proven SQLite BaaS today, use it. Fieldstone's edge is breadth and ownership: Postgres + GraphQL + gRPC + WASM plugins + a real job queue + cache/ratelimit as first-class, and it's your code to extend. You trade community maturity for control and a wider built-in surface.
Where Fieldstone genuinely improves: (1) Postgres → multi-instance scale + your DBA tooling; (2) GraphQL/gRPC for typed, efficient clients; (3) WASM plugins → custom server logic in any language, sandboxed, safer than an embedded JS VM; (4) a proper background-job engine instead of rolling your own.

Fieldstone as an in-house backend

Because it's one self-hostable binary with Postgres + auth + realtime + jobs + storage, Fieldstone can be the backend you own and run — no per-seat SaaS bill, no vendor lock-in, and (the big one for you) data residency in India.

100xLongevity

Today it's on Supabase (US-managed). Fieldstone is the in-house alternative when DPDP data residency, cost control, or sovereignty matter:

  • Postgres + auth + storage → the same primitives Supabase gives, on your infra.
  • Realtime hub → live biomarker / score / consult-status updates to the app.
  • Job queue → the score-compute + signal crons (replaces pg_cron + edge functions).
  • WASM plugins → the scoring engine + OCR post-processing, sandboxed.
  • Storage → lab-report / OCR file storage with signed URLs.

Trade-off: you operate it (backups, uptime) vs Supabase doing that for you. Sensible path: keep Supabase for the MVP, pilot Fieldstone for India-residency / a self-hosted tier.
